# 背景 / 目标

服务启动后异常退出，检查发现 `.run/funnel-18888.pid` 记录成了 Chrome Helper 的 PID，而不是 Uvicorn 服务进程。根因是启动脚本用 `lsof -ti tcp:18888` 未限定监听态，可能抓到浏览器客户端连接进程。

目标：修复启动/停止脚本的 PID 识别，避免误判服务已运行或误杀非服务进程，并恢复 18888 服务可访问。

# 任务分解（checklist）

- [ ] 修改 `start.sh`，只使用 `LISTEN` 状态的端口进程作为服务 PID。
- [ ] 修改 `start.sh`，读取 PID 文件时校验该 PID 是否真的是服务进程。
- [ ] 修改 `stop.sh`，避免根据错误 PID 文件杀掉非 Uvicorn 进程。
- [ ] 执行脚本语法校验。
- [ ] 执行 `./restart.sh` 并验证首页返回 200。
- [ ] 提交并推送修复。

# 验收标准

- `.run/funnel-18888.pid` 记录的是实际监听 18888 的服务 PID。
- `./start.sh` 不会把浏览器或其他客户端连接误认为服务。
- `./stop.sh` 不会杀掉 PID 文件中误写入的非服务进程。
- `http://127.0.0.1:18888/` 返回 200。

# 风险与回滚

- 风险：本次修改影响服务启动/停止脚本，若判断条件过严可能导致旧服务无法自动停止。
- 回滚：恢复 `start.sh` / `stop.sh` 到上一个提交版本，并手动清理端口进程后启动。

# 关键决策记录

- 端口 PID 只接受 `TCP LISTEN` 状态进程。
- PID 文件只作为候选，必须同时满足服务命令或监听端口校验。
