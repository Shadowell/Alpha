# 背景 / 目标

数据中心增量同步当前由前端按日期循环调用单日接口，导致请求多、交互慢，也容易在页面侧表现为卡住。需要改为一次请求提交整个日期范围，由后端批量入队并顺序执行。

# 任务分解（checklist）

- [ ] 新增后端批量增量同步接口，接收 `start_date` / `end_date`。
- [ ] 服务层新增批量入队方法，避免前端逐日调用。
- [ ] 前端增量同步按钮改为一次调用批量接口。
- [ ] README 更新 K 线缓存 API 列表。
- [ ] 执行 Python/JS/脚本基础校验。
- [ ] 执行 `./restart.sh` 并验证服务可访问。

# 验收标准

- 选择日期范围后，前端只发一次批量同步请求。
- 后端返回 202 Accepted，显示提交的交易日数量和队列长度。
- 后端仍按日期顺序执行，避免并发写库冲突。
- 服务重启后首页与状态接口正常返回。

# 风险与回滚

- 风险：批量提交过多日期会排队较长，但不会阻塞页面请求。
- 回滚：恢复前端逐日调用 `/incremental-sync`，保留后端接口不使用。

# 关键决策记录

- 批量接口只负责提交队列，不等待实际同步完成。
- 日期范围在后端按工作日粗过滤，最终是否交易日仍由原单日同步逻辑判断。
