# 背景 / 目标

数据中心补数任务在执行过程中停留在 `running`，最新任务停在 `320/2601`，服务进程进入 `UEs` 并持有大量 `market_kline.db` / WAL 句柄。目标是降低补数期间的 SQLite 写入频率与 WAL 压力，并让中断任务在后续状态查询中能正确收尾。

# 任务分解（checklist）

- [x] 将 K 线补数的逐条写库改为每批次统一写入。
- [x] 将任务明细、任务进度、同步状态更新合并到批量事务中。
- [x] 补齐历史 `running` 任务的中断收尾逻辑。
- [x] 增加针对批量写入和中断收尾的测试。
- [x] 执行可行的测试验证。

# 验收标准

- 补数抓取阶段不再每处理一只股票就打开 SQLite 连接并提交事务。
- 每批抓取完成后，K 线、任务明细、任务进度和同步状态在同一批次内落库。
- 服务重启或任务队列丢失后，状态接口能把陈旧 `running` 任务标记为失败。
- 相关单元测试通过。

# 风险与回滚

- 当前机器仍存在 `UEs` 进程，无法可靠执行服务重启验证；代码验证以单元测试为主。
- 若批量事务引入异常，可回滚到逐条写入逻辑，但会恢复原有 WAL 压力。

# 关键决策记录

- 优先改写服务层批处理，不改动外部 Akshare 接口形态。
- 保留现有任务队列 API，避免影响前端调用。
