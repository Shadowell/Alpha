# 背景 / 目标

数据中心执行全量补缺时页面停在进度中不动，服务进程出现 `UEs` 状态，状态接口也会超时。当前同步接口会一直等待完整补缺任务结束，且单只股票远端抓取没有业务层限时，远端 AkShare/网络断连时容易拖慢整个服务。

目标：让数据中心同步任务不阻塞页面请求，提升进度刷新频率，并为单只股票远端抓取增加超时保护，降低同步卡死概率。

# 任务分解（checklist）

- [ ] 同步接口改为提交后台任务后立即返回。
- [ ] 服务层增加后台任务状态保护，避免重复启动多个同步。
- [ ] 单只股票 K 线抓取增加限时，超时后记失败并继续下一个。
- [ ] 进度落库频率从每 50 只调整为更细粒度。
- [ ] 前端按钮文案改为任务已提交，依靠轮询刷新状态。
- [ ] 执行 JS/Python 基础校验。
- [ ] 执行 `./restart.sh` 并验证服务可访问。

# 验收标准

- 点击“全量补缺/增量同步”后请求快速返回，不再等完整同步结束。
- 数据中心进度至少每 10 只或任务结束时更新。
- 单只股票远端抓取超时不会卡住整批任务。
- 状态接口 `/api/jobs/kline-cache/status` 可在同步期间快速返回。
- 服务重启后首页返回 200。

# 风险与回滚

- 风险：过短超时会增加失败数，但能保证服务可用性。
- 回滚：恢复同步接口为阻塞执行，并移除单只抓取超时。

# 关键决策记录

- 页面交互优先保证可用性，同步任务长耗时全部后台执行。
- 超时股票记录为失败，后续可再次补缺，而不是卡住当前任务。
