# 背景 / 目标

现有“热门个股”面板虽然底层 `AkshareDataProvider.get_hot_stocks()` 已优先走实时接口，但 `FunnelService` 会把热门股列表持久化到本地状态，并在部分场景直接返回旧快照，盘后还会冻结刷新。这与“热门个股应每 5 分钟从接口获取一次，而不是主要依赖本地数据库/本地持久化”的要求不一致。

本次目标是把热门个股改为独立的 5 分钟接口刷新逻辑：

- 热门个股接口默认每 5 分钟重新拉取一次上游接口。
- 本地状态仅作为最近一次成功结果的兜底，不再作为主数据源。
- 漏斗三池冻结状态不再阻断热门个股刷新。
- 增加回归测试，覆盖 stale cache 与 frozen 场景。

# 任务分解（checklist）

- [ ] 为热门个股增加独立更新时间字段，不再复用漏斗全局 `updated_at`
- [ ] 调整 `FunnelService.get_hot_stocks()`，当缓存超过 5 分钟时强制刷新
- [ ] 调整 `FunnelService.tick()`，后台循环按 5 分钟节奏刷新热门个股
- [ ] 保留接口失败时的最近一次成功结果兜底
- [ ] 增加单元测试，验证 frozen / stale 场景
- [ ] 执行相关测试与 `./restart.sh`
- [ ] 提交并推送到 `origin/main`

# 验收标准

- `/api/market/hot-stocks` 返回的结果在 5 分钟缓存窗口后会重新命中上游接口
- 即使 `frozen=true`，热门个股仍可按 5 分钟节奏刷新
- 接口失败时不会清空已有热门个股列表
- 新增测试通过，且不影响现有漏斗/行情相关接口

# 风险与回滚

- 风险：若上游接口抖动，5 分钟刷新可能导致日志中出现更多失败记录
- 控制：继续使用最近一次成功结果作为兜底，不把失败结果写回空列表
- 回滚：可回退到本次提交前版本，恢复原有基于本地状态的热门股返回方式

# 关键决策记录

- 保留 `AkshareDataProvider` 的 300 秒缓存，避免前端高频轮询直接打穿上游接口
- 只改“热门个股”刷新语义，不动现有三池、热门概念与盘中打分主流程
