# 背景 / 目标

当前系统已经具备“热门个股实时接口获取”能力，但还缺少一个真正可用的“热门股票智能分析模块”：

- 以实时热门股接口为输入，而不是本地数据库排行；
- 拉取热门前 20 股票；
- 对每只股票逐个分析并输出可解释评分；
- 按评分拆分为候选池 / 重点关注池 / 买入候选池三池；
- 提供 FastAPI 接口、前端页面、自动刷新和持久化快照。

本次目标是在 Alpha 现有架构内，把该模块完整落地为一个独立页面与后台服务，复用现有 `KronosPredictService`、K 线缓存和 SQLite 状态存储能力。

# 任务分解（checklist）

- [ ] 设计热门股票智能分析服务的数据结构、配置项和持久化键
- [ ] 实现热门前 20 股票抓取、逐股分析、解释性评分和三池分配
- [ ] 接入自动刷新调度，默认每 5 分钟运行一次
- [ ] 暴露 `snapshot / run / config` FastAPI 接口
- [ ] 新增前端 Tab、三池卡片渲染、详情弹窗跳转
- [ ] 增加服务单测与 API 回归覆盖
- [ ] 更新 `README.md` 的模块说明、界面章节和接口列表
- [ ] 执行测试、`./restart.sh`、提交并推送 `origin/main`

# 验收标准

- 页面能看到热门前 20 股票的智能分析结果与三池拆分
- 后端接口返回 `running/progress/config/pools/meta` 完整快照
- 每只股票都有分数、分数组成、核心指标和简要分析说明
- 默认每 5 分钟自动刷新一次，且可手动触发
- 测试通过，服务重启后接口可访问

# 风险与回滚

- 风险：Kronos 首次加载慢，导致第一次全量扫描耗时较长
- 控制：服务保留进度与最近一次快照，失败时不覆盖已有成功结果
- 风险：部分热门股缺少足够 K 线历史
- 控制：缺数据个股跳过并计入 `failed_count`，不中断全任务
- 回滚：删除本次新增服务、接口和前端入口，恢复到仅有热门榜展示的版本

# 关键决策记录

- 模块输入固定为实时热门股接口 `top_n=20`
- 分析引擎优先复用 `Kronos` 预测与本地历史 K 线，不额外引入新的 LLM 链路
- 页面形态沿用现有 `predict / graphic` 三池布局，降低接入成本并保持统一体验
